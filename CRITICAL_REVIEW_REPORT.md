# Критичный обзор текущей кодовой базы MedApp (сервер)

## 1. Область анализа

Проверена серверная часть `src/MedAppServer` (контроллеры, сервисы, модели и конфигурация). Анализ проводится с учетом того, что это отдельная серверная реализация и часть требований мобильного приложения может быть намеренно опущена.

## 2. Что реализовано

- **Базовые сущности**: `User`, `MedKit`, `Drug`, `Using`, а также справочник `VidalDrug`.
- **Основные API**:
  - Регистрация и выдача JWT (`/auth/register`, `/auth/login`).
  - CRUD для аптечек и совместный доступ через share-ключ.
  - CRUD для препаратов и перенос между аптечками.
  - План лечения как резерв количества (planned amount) и фиксация приема.
  - Fuzzy-search по каталогу препаратов.
- **Приватность**: персональные данные пользователей не хранятся, идентификация по сгенерированному ключу.
- **Инфраструктура**: Spring Boot + JPA/Hibernate, PostgreSQL, OpenAPI (Swagger).

## 3. Существенные расхождения с ТЗ

> Ниже перечислены ключевые отличия от требований ТЗ. Часть из них может быть допустима для серверной части, но их важно явно учитывать при мердже.

### 3.1 Архитектурные отличия

- В ТЗ заявлен **Ktor + Exposed**, по факту используется **Spring Boot + JPA**. Это влияет на технологическую совместимость и набор зависимостей.

### 3.2 Модель данных

Серверные модели значительно проще и не содержат всех обязательных атрибутов из ТЗ. Например:

- **MedKit**: отсутствуют обязательные поля *название* и *место хранения*.
- **Drug**: отсутствуют *срок годности*, *однократная доза*, *дата покупки*, *цена*, *заметки*, *дата вскрытия упаковки* и т.д.
- **Using / план лечения**: отсутствуют даты начала/окончания, расписание по дням недели, интервалы приема, напоминания.

Это допустимо, если сервер не является единственным источником данных, но несоответствие нужно учитывать при синхронизации.

### 3.3 Функциональные пробелы

- **Поиск** реализован только по справочнику препаратов, но нет поиска по пользовательским аптечкам.
- **Фильтрация и сортировка** отсутствуют на стороне сервера.
- **Просроченные препараты**: нет полей и логики для учета сроков годности и уведомлений.
- **Аналитика и статистика** не реализованы.
- **Сканирование штрихкодов** и интеграция с внешним API не представлены.
- **Ролевое управление общей аптечкой** не предусмотрено (соответствует ТЗ), но нет UI-предупреждений о необратимости доступа (важно для клиента).

## 4. Риски и потенциальные проблемы

- **Схема данных** не поддерживает обязательные поля ТЗ → клиентам придется хранить их локально и обеспечивать логику отдельно.
- **План лечения** хранится только как `plannedAmount` и может конфликтовать с ожиданиями клиента (нет расписаний, дат и напоминаний).
- **Различие технологий** (Spring Boot вместо Ktor) может усложнить поддержку и соответствие требованиям курса.

## 5. Рекомендации

1. Зафиксировать, какие поля обязательны на сервере, а какие остаются на клиенте (возможно в отдельной спецификации).
2. Определить минимальный контракт синхронизации: какие поля по препаратам должны быть общими между клиентом и сервером.
3. Решить, нужно ли адаптировать сервер к требованиям ТЗ по технологии (Ktor + Exposed), либо зафиксировать исключение.
4. Если аналитика и уведомления остаются на клиенте — описать это в архитектуре и документации.
