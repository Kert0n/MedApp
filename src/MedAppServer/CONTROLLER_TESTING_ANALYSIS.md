# Анализ Целей Тестирования Контроллеров

## 1. Симуляция Обычного Использования

### Типичные Сценарии Пользователя

#### DrugController
1. **Создание лекарства** - Пользователь добавляет новое лекарство в аптечку
2. **Просмотр лекарства** - Получение информации о лекарстве
3. **Обновление лекарства** - Изменение количества, срока годности
4. **Удаление лекарства** - Удаление из аптечки
5. **Потребление лекарства** - Уменьшение количества при приёме
6. **Перемещение лекарства** - Перенос между аптечками
7. **Поиск по шаблону** - Fuzzy search по названию лекарства
8. **Проверка доступного количества** - С учётом запланированного использования

#### MedKitController
1. **Создание аптечки** - Новая личная аптечка
2. **Просмотр аптечки** - Получение информации об аптечке
3. **Список всех аптечек** - Все доступные пользователю аптечки
4. **Предоставление доступа** - Добавление другого пользователя
5. **Выход из общей аптечки** - Удаление себя из участников
6. **Удаление аптечки** - С переносом или без переноса лекарств

#### TreatmentPlanController
1. **Создание плана лечения** - Планирование приёма лекарства
2. **Просмотр планов** - Все планы пользователя
3. **Обновление плана** - Изменение дозировки/периодичности
4. **Фиксация приёма** - Отметка о принятом лекарстве
5. **Удаление плана** - Отмена курса лечения

#### UserController
1. **Получение данных пользователя** - Вся информация с аптечками

#### AuthController
1. **Регистрация** - Создание нового пользователя
2. **Получение токена** - Аутентификация

## 2. Крайние Случаи (Edge Cases)

### Граничные Значения
- **Нулевое количество** - Лекарство закончилось
- **Отрицательные значения** - Попытка установить отрицательное количество
- **Максимальные значения** - Очень большие числа
- **Пустые строки** - Пустое название, описание
- **Специальные символы** - В названиях, описаниях
- **Длинные строки** - Превышение лимита символов

### Конфликты Данных
- **Конкурентное обновление** - Два пользователя обновляют одно лекарство
- **Удаление во время использования** - Лекарство удалено, но на него есть ссылка
- **Превышение доступного количества** - Попытка запланировать больше, чем есть
- **Круговые зависимости** - Перенос между аптечками

### Авторизация
- **Доступ к чужой аптечке** - Попытка доступа без прав
- **Доступ к чужому лекарству** - В недоступной аптечке
- **Истёкший токен** - Неавторизованный запрос
- **Отсутствующий пользователь** - User ID не существует

### Несуществующие Ресурсы
- **Несуществующее лекарство** - Drug ID не найден
- **Несуществующая аптечка** - MedKit ID не найден
- **Несуществующий план** - Treatment plan не найден
- **Несуществующий шаблон** - Template ID не найден

## 3. Типичные Баги в REST Контроллерах

### Проблемы Валидации
1. **Пропущенная валидация входных данных**
   - Отсутствие @Valid на RequestBody
   - Недостаточные constraints на DTO
   - Неправильная обработка null значений

2. **Неправильные HTTP статус-коды**
   - 200 вместо 201 для создания
   - 200 вместо 204 для удаления
   - 500 вместо 400 для невалидных данных
   - 500 вместо 404 для несуществующих ресурсов
   - 500 вместо 403 для проблем авторизации

### Проблемы Безопасности
1. **Отсутствие проверки авторизации**
   - Проверка Authentication, но не проверка прав на ресурс
   - Обход проверки через прямой вызов сервиса

2. **Утечка информации**
   - Раскрытие чувствительных данных в ошибках
   - Возврат больше данных, чем нужно
   - Информация о существовании ресурсов

### Проблемы Обработки Исключений
1. **Неправильный перехват исключений**
   - Ловля всех Exception вместо конкретных
   - Потеря stack trace
   - Некорректные сообщения об ошибках

2. **Отсутствие обработки edge cases**
   - Division by zero
   - Null pointer exceptions
   - Concurrent modification exceptions

### Проблемы Транзакций
1. **Неправильные границы транзакций**
   - Транзакция не откатывается при ошибке
   - Частичное сохранение данных

2. **Проблемы с ленивой загрузкой (Lazy Loading)**
   - LazyInitializationException после закрытия сессии
   - N+1 query problem

### Проблемы с Данными
1. **Несогласованность данных**
   - Orphaned records после удаления
   - Неправильные каскадные операции
   - Нарушение referential integrity

2. **Проблемы с конкурентным доступом**
   - Lost updates
   - Dirty reads
   - Non-repeatable reads

## 4. Матрица Тестов для Каждого Endpoint

### Категории Тестов (минимум 5 на каждый endpoint)

1. **Happy Path (Позитивный сценарий)**
   - Корректные данные
   - Авторизованный пользователь
   - Все условия выполнены

2. **Authorization (Авторизация)**
   - Неавторизованный запрос (без токена)
   - Доступ к чужому ресурсу (forbidden)
   - Несуществующий пользователь

3. **Validation (Валидация)**
   - Невалидные данные в body
   - Отсутствующие обязательные поля
   - Неправильный формат данных
   - Граничные значения

4. **Not Found (Не найдено)**
   - Несуществующий ID ресурса
   - Удалённый ресурс

5. **Business Logic (Бизнес-логика)**
   - Конфликты данных
   - Нарушение бизнес-правил
   - Edge cases специфичные для метода

## 5. Специфичные Тесты по Контроллерам

### DrugController (9 endpoints × 5+ тестов = 45+ тестов)

#### GET /drug/{id}
1. Happy: Получение существующего лекарства
2. Not Found: Несуществующий drug ID
3. Forbidden: Доступ к лекарству из чужой аптечки
4. Unauthorized: Запрос без аутентификации
5. Edge: Лекарство с нулевым количеством

#### POST /drug
1. Happy: Создание нового лекарства с валидными данными
2. Validation: Отсутствует обязательное поле (name)
3. Validation: Отрицательное quantity
4. Forbidden: MedKit ID недоступен пользователю
5. Not Found: Несуществующий medKitId
6. Edge: Создание с минимальными данными

#### PUT /drug/{id}
1. Happy: Обновление всех полей
2. Not Found: Несуществующий drug ID
3. Forbidden: Обновление чужого лекарства
4. Validation: Отрицательное количество
5. Edge: Обновление только одного поля

#### DELETE /drug/{id}
1. Happy: Успешное удаление (204)
2. Not Found: Удаление несуществующего
3. Forbidden: Удаление чужого лекарства
4. Edge: Удаление при наличии treatment plan

#### GET /drug/quantity/{id}
1. Happy: Получение информации о количестве
2. Not Found: Несуществующий drug
3. Forbidden: Чужое лекарство
4. Business Logic: С активным treatment plan (planned > 0)
5. Edge: Нулевое количество

#### PUT /drug/consume/{id}
1. Happy: Потребление валидного количества
2. Validation: Отрицательное quantity
3. Validation: Попытка потребить больше, чем есть
4. Not Found: Несуществующий drug
5. Forbidden: Чужое лекарство
6. Edge: Потребление всего доступного (0 в остатке)

#### PUT /drug/move/{id}
1. Happy: Перенос в другую аптечку
2. Not Found: Несуществующий drug
3. Not Found: Несуществующая целевая аптечка
4. Forbidden: Целевая аптечка недоступна
5. Forbidden: Исходное лекарство недоступно
6. Edge: Перенос с treatment plans

#### GET /drug/template/search
1. Happy: Поиск с результатами
2. Happy: Поиск без результатов
3. Edge: Пустой searchTerm
4. Edge: Limit = 0
5. Edge: Очень большой limit
6. Edge: Спецсимволы в поиске

#### GET /drug/template/{id}
1. Happy: Получение существующего шаблона
2. Not Found: Несуществующий template ID
3. Edge: Шаблон с минимальными данными

### MedKitController (6 endpoints × 6+ тестов = 36+ тестов)

#### POST /med-kit
1. Happy: Создание новой аптечки (201)
2. Edge: Множественное создание
3. Unauthorized: Без аутентификации

#### GET /med-kit/{id}
1. Happy: Получение своей аптечки
2. Happy: Получение общей аптечки (где пользователь участник)
3. Not Found: Несуществующая аптечка
4. Forbidden: Чужая аптечка

#### GET /med-kit
1. Happy: Получение списка всех аптечек
2. Edge: У пользователя нет аптечек
3. Edge: Только личные аптечки
4. Edge: Только общие аптечки
5. Edge: Смешанные личные и общие

#### POST /med-kit/{id}/share
1. Happy: Добавление пользователя в аптечку
2. Not Found: Несуществующая аптечка
3. Not Found: Несуществующий userId для добавления
4. Forbidden: Добавление в чужую аптечку
5. Edge: Повторное добавление того же пользователя
6. Validation: Отсутствует userId в body

#### DELETE /med-kit/{id}/leave
1. Happy: Выход из общей аптечки (204)
2. Not Found: Несуществующая аптечка
3. Business Logic: Выход последнего пользователя (аптечка удаляется)
4. Business Logic: Выход с удалением своих treatment plans

#### DELETE /med-kit/{id}
1. Happy: Удаление без переноса лекарств
2. Happy: Удаление с переносом в другую аптечку
3. Not Found: Несуществующая аптечка
4. Not Found: Несуществующая целевая аптечка для переноса
5. Forbidden: Удаление чужой аптечки
6. Forbidden: Перенос в недоступную аптечку
7. Business Logic: Общая аптечка - только выход пользователя

### TreatmentPlanController (6 endpoints × 6+ тестов = 36+ тестов)

#### GET /treatment-plan
1. Happy: Получение всех планов пользователя
2. Edge: У пользователя нет планов
3. Edge: Множественные планы

#### GET /treatment-plan/drug/{drugId}
1. Happy: Получение плана для конкретного лекарства
2. Not Found: План не существует (null)
3. Not Found: Несуществующий drug
4. Forbidden: Лекарство из чужой аптечки

#### POST /treatment-plan
1. Happy: Создание плана лечения
2. Validation: plannedAmount <= 0
3. Validation: Отсутствует drugId
4. Not Found: Несуществующий drug
5. Forbidden: Drug из чужой аптечки
6. Business Logic: plannedAmount превышает available quantity
7. Edge: План уже существует для этого drug

#### PUT /treatment-plan/drug/{drugId}
1. Happy: Обновление существующего плана
2. Validation: Отрицательный plannedAmount
3. Not Found: План не существует
4. Forbidden: Чужое лекарство
5. Business Logic: Новое количество превышает доступное

#### POST /treatment-plan/drug/{drugId}/intake
1. Happy: Фиксация приёма
2. Validation: Отрицательный quantityConsumed
3. Validation: Превышает planned amount
4. Not Found: План не существует
5. Forbidden: Чужой план

#### DELETE /treatment-plan/drug/{drugId}
1. Happy: Удаление плана (204)
2. Not Found: План не существует
3. Forbidden: Чужой план
4. Business Logic: Количество лекарства не меняется

### UserController (1 endpoint × 5+ тестов = 5+ тестов)

#### GET /user
1. Happy: Получение данных пользователя с аптечками
2. Edge: Пользователь без аптечек
3. Edge: Пользователь с множественными аптечками
4. Edge: Общие и личные аптечки
5. Unauthorized: Без аутентификации

### AuthController (2 endpoints × 7+ тестов = 14+ тестов)

#### POST /auth/register
1. Happy: Регистрация с корректным секретом
2. Unauthorized: Неправильный секрет
3. Edge: Множественные регистрации
4. Edge: Уникальность login/key

#### GET /auth/login
1. Happy: Получение JWT токена
2. Happy: Токен содержит корректные claims
3. Happy: Токен имеет корректный срок действия
4. Unauthorized: Без аутентификации
5. Edge: Повторный запрос токена
6. Edge: Валидация структуры токена

## 6. Инструменты Тестирования

### Framework: Spring Boot Test + MockMvc
- `@WebMvcTest` - изолированное тестирование контроллера
- `@MockBean` - моки для сервисов
- `MockMvc` - имитация HTTP запросов
- Mockito Kotlin - для stubbing и verification

### Assertions
- HTTP status codes (2xx, 4xx, 5xx)
- Response body structure и содержимое
- Headers (Content-Type, Location для 201)
- Вызовы методов сервиса (verify)

### Test Data
- TestDataBuilders - для создания тестовых объектов
- Фикстуры - стандартные наборы данных
- UUID - использование предсказуемых ID

## 7. Ожидаемый Результат

### Покрытие
- **Минимум**: 5 тестов на каждый endpoint
- **Общее количество**: ~140+ тестов для контроллеров
- **Типы покрытия**: Happy path, validation, authorization, edge cases, business logic

### Качество
- Все тесты изолированы (не зависят друг от друга)
- Используют моки (быстрое выполнение)
- Проверяют корректное поведение и обработку ошибок
- Документируют ожидаемое поведение API

### Интеграция с CI/CD
- Все тесты проходят автоматически
- Быстрое выполнение (<30 секунд)
- Высокая стабильность (нет flaky tests)
