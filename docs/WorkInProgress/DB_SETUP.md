# Applying the Data Model to PostgreSQL

## Current State

The data model is split into two groups:

### 1. Parsed (template) data — already in the database
These entities represent drug catalog data parsed from external sources (e.g. Vidal).
They serve as **templates** that users can reference or override with their own data.

| Entity         | Table             | Description                       |
|----------------|-------------------|-----------------------------------|
| `VidalDrug`    | `drugs`           | Parsed drug catalog entries       |
| `FormType`     | `form_types`      | Drug form type lookup (e.g. tablet, capsule) |
| `QuantityUnit` | `quantity_units`   | Quantity unit lookup (e.g. mg, ml) |

### 2. User data — waiting to be mapped
These entities represent user-specific data and are now fully annotated for JPA/Hibernate.

| Entity   | Table            | Description                                     |
|----------|------------------|-------------------------------------------------|
| `Drug`   | `user_drugs`     | User's own drug entries (can override parsed data) |
| `User`   | `users`          | Application users                               |
| `Medbox` | `medboxes`       | Medicine boxes shared between users              |
| `Uses`   | `uses`           | Junction: which user uses which drug, with pattern |

Join table (auto-generated by JPA):

| Table            | Description                         |
|------------------|-------------------------------------|
| `user_medboxes`  | Many-to-many join between users and medboxes |

## How to Apply to PostgreSQL

### Option A: Hibernate auto-DDL (development only)

Add to `application.properties`:

```properties
spring.jpa.hibernate.ddl-auto=update
```

This tells Hibernate to automatically create/update tables based on entity annotations.

- `update` — creates missing tables and columns, never drops anything.
- `create` — drops and recreates all tables on every startup (**data loss**).
- `validate` — only checks that the schema matches entities; fails if it doesn't.
- `none` — does nothing (use this in production with Flyway/Liquibase).

### Option B: Flyway migrations (recommended for production)

1. Add the Flyway dependency to `build.gradle.kts`:

```kotlin
implementation("org.flywaydb:flyway-core")
implementation("org.flywaydb:flyway-database-postgresql")
```

2. Create the migration directory:

```
src/main/resources/db/migration/
```

3. Create migration files. Example `V1__create_user_tables.sql`:

```sql
CREATE TABLE users (
    id UUID PRIMARY KEY,
    hashed_key VARCHAR(255) NOT NULL
);

CREATE UNIQUE INDEX ix_users_hashed_key ON users (hashed_key);

CREATE TABLE medboxes (
    id UUID PRIMARY KEY
);

CREATE TABLE user_medboxes (
    user_id UUID NOT NULL REFERENCES users(id),
    medbox_id UUID NOT NULL REFERENCES medboxes(id),
    PRIMARY KEY (user_id, medbox_id)
);

CREATE TABLE user_drugs (
    id UUID PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    quantity DOUBLE PRECISION NOT NULL,
    quantity_unit VARCHAR(255) NOT NULL,
    form_type VARCHAR(255),
    category VARCHAR(255),
    manufacturer VARCHAR(255),
    country VARCHAR(255),
    description TEXT
);

CREATE INDEX ix_user_drugs_name ON user_drugs (name);

CREATE TABLE uses (
    user_id UUID NOT NULL REFERENCES users(id),
    drug_id UUID NOT NULL REFERENCES user_drugs(id),
    pattern VARCHAR(255) NOT NULL,
    PRIMARY KEY (user_id, drug_id)
);
```

4. Set `ddl-auto=validate` so Hibernate only validates (Flyway manages the schema):

```properties
spring.jpa.hibernate.ddl-auto=validate
```

### Quick Start (development)

1. Start PostgreSQL via Docker Compose:

```bash
docker compose up -d
```

2. Add these properties to `application.properties`:

```properties
spring.jpa.hibernate.ddl-auto=update
spring.jpa.show-sql=true
```

3. Run the application — Hibernate will create all missing tables automatically.

## Index Summary

### Parsed tables (already in DB)
- `drugs.name` — search drugs by name
- `drugs.form_type_id` — FK join optimization
- `drugs.quantity_unit_id` — FK join optimization
- `drugs.active_substance` — search by active substance
- `drugs.manufacturer` — search by manufacturer

### User tables (new)
- `users.hashed_key` (unique) — authentication lookups
- `user_drugs.name` — search user drugs by name
- `user_medboxes` — composite PK on (user_id, medbox_id) acts as index
- `uses` — composite PK on (user_id, drug_id) acts as index
